# 终端音游 屏幕刷新技术报告

## 一、概述

本项目采用 **ANSI VT100 转义序列 + 内存缓冲区单次写入 + 精确时钟定时** 的方案实现终端游戏的无闪烁高帧率渲染。理论帧率 125fps（8ms/帧），实际帧率受终端渲染能力和系统调度精度影响。

---

## 二、为什么不能用 system("cls")

最朴素的终端刷新方式是每帧 `system("cls")` 清屏再重新打印。问题：

| 缺陷 | 原因 |
|------|------|
| 明显白屏闪烁 | cls 清空整个屏幕缓冲区，终端重绘前有空白期 |
| 帧率上限低 | system() 启动子进程开销 3~10ms，加上输出时间极限约 30fps |
| 画面撕裂 | 清屏和打印之间终端会渲染中间状态 |
| 滚动历史污染 | 每次清屏会留下空行记录 |

---

## 三、本项目的方案

### 核心思路

**不清屏，把光标移回左上角，用新内容直接覆盖旧内容，整帧一次性写入。**

### 3.1 备用屏幕缓冲区（Alternate Screen Buffer）

```c
printf("\033[?1049h");  // 进入备用屏幕
printf("\033[?1049l");  // 退出备用屏幕
```

终端有两个屏幕缓冲区：
- **主屏幕**：日常命令行，有滚动历史
- **备用屏幕**：全屏程序使用（vim、less、htop 都用这个）

游戏开始时切到备用屏幕，结束时切回主屏幕。效果：
- 游戏画面完全隔离，不污染终端滚动历史
- 退出后终端恢复到游戏开始前的状态
- 用户无法上滑看到游戏渲染的中间帧

### 3.2 光标归位（Cursor Home）

```c
snprintf(buf + len, ..., "\033[H");
```

`\033[H` = CUP (Cursor Position) 命令，把光标移动到第1行第1列。

与 cls 的区别：
- cls 先擦除所有内容再写 → 有空白期
- `\033[H` 只移光标不擦除 → 新内容直接覆盖旧内容，无空白期

### 3.3 行尾清除（Erase to End of Line）

```c
snprintf(buf + len, ..., " GOOD:%-4d  BAD:%-4d  MISS:%-4d\033[K\n", good, bad, miss);
```

`\033[K` = EL (Erase in Line) 命令，清除从光标当前位置到行尾。

作用场景：
- 新内容比旧内容短 → 清除残留字符
- 新内容等长或更长 → 无副作用

不用 `\033[2K`（清除整行）是因为那样会先擦后写，行首也会闪。

### 3.4 屏幕底部清除

```c
snprintf(buf + len, ..., "\033[J");
```

`\033[J` = ED (Erase in Display) 命令，清除从光标到屏幕底部所有内容。

放在每帧最后一行之后。确保：如果上一帧比当前帧多了几行（比如窗口缩小），多余内容会被擦除。

### 3.5 内存缓冲区 + 单次系统调用写入

```c
// 分配缓冲区
vector<char> buf(buf_size);
int len = 0;

// 所有内容拼接到 buf（不做任何 I/O）
len += snprintf(buf.data() + len, ..., "\033[H");
len += snprintf(buf.data() + len, ..., "... 第一行 ...\033[K\n");
len += snprintf(buf.data() + len, ..., "... 第二行 ...\033[K\n");
// ... 更多行 ...
len += snprintf(buf.data() + len, ..., "\033[J");

// 一次性写出整帧
#ifdef _WIN32
WriteConsoleA(hOut, buf.data(), (DWORD)len, &written, NULL);
#else
::write(STDOUT_FILENO, buf.data(), len);
#endif
```

为什么不用 cout/printf 逐行输出：

| 方式 | 系统调用次数 | 撕裂风险 | 帧率影响 |
|------|-------------|---------|---------|
| cout 逐行 | ~20次/帧 | 高 | 明显降低 |
| printf 逐行 | ~20次/帧（带缓冲稍好） | 中 | 有影响 |
| buf + 单次 write | 1次/帧 | 极低 | 几乎无影响 |

单次 `WriteConsoleA` / `write` 将整帧作为一个原子操作交给内核。终端模拟器收到完整数据后一次性渲染，不会出现"画了一半"的中间状态。

### 3.6 精确帧定时（Fixed Timestep）

```c
auto next_frame = std::chrono::steady_clock::now();

for(int i = 0; i < total_frames; i++)
{
    // ... 帧逻辑 + 渲染 ...

    next_frame += std::chrono::milliseconds(8);
    std::this_thread::sleep_until(next_frame);
}
```

#### 为什么不用 Sleep(8)

`Sleep(8)` / `usleep(8000)` 是"从现在开始至少等8ms"。实际流程：

```
帧逻辑耗时 1ms + Sleep(8ms) = 实际帧间隔 9ms → 实际帧率 111fps
帧逻辑耗时 2ms + Sleep(8ms) = 实际帧间隔 10ms → 实际帧率 100fps
```

帧率会随负载波动，长时间运行误差累积。

#### sleep_until 的工作原理

`sleep_until` 是"睡到指定绝对时刻"：

```
帧0开始时刻：T
帧0目标：T + 8ms  → 帧逻辑花 1ms → 实际只睡 7ms
帧1目标：T + 16ms → 帧逻辑花 2ms → 实际只睡 6ms
帧2目标：T + 24ms → 帧逻辑花 1ms → 实际只睡 7ms
```

每帧的目标时刻是固定的 `起始 + N×8ms`，帧内逻辑耗时被自动补偿。帧率严格锁定。

### 3.7 隐藏光标

```c
printf("\033[?25l");  // 隐藏
printf("\033[?25h");  // 显示
```

如果不隐藏光标，每帧写入时光标会在屏幕上快速跳动，产生明显的闪烁方块。隐藏后画面干净。

### 3.8 Windows VT100 支持

Windows 10+ 的 cmd / Windows Terminal 支持 ANSI 转义，但默认关闭。需要手动开启：

```c
HANDLE hOut = GetStdHandle(STD_OUTPUT_HANDLE);
DWORD mode = 0;
GetConsoleMode(hOut, &mode);
SetConsoleMode(hOut, mode | ENABLE_VIRTUAL_TERMINAL_PROCESSING);
```

开启后所有 `\033[...` 序列都能正确解析。

## 四、帧率分析

### 4.1 理论帧率

| 参数 | 值 |
|------|------|
| 帧间隔设定 | 8ms |
| 理论帧率 | 1000 / 8 = **125 fps** |
| 每秒渲染次数 | 125 次 |

### 4.2 每帧时间预算分配

8ms 预算内需要完成：

| 阶段 | 预估耗时 | 说明 |
|------|---------|------|
| 键盘输入读取 | < 0.01ms | _kbhit + _getch，内核缓冲区读取 |
| 判定逻辑计算 | < 0.05ms | 遍历音符数组，简单算术 |
| 帧画面拼接 | 0.05~0.2ms | snprintf 拼接 ~500 字节 |
| 系统调用写入 | 0.1~1ms | WriteConsoleA 写 ~500 字节到终端 |
| 终端渲染 | 0.5~3ms | 终端模拟器解析 ANSI 并绘制到屏幕 |
| sleep_until 等待 | 剩余时间 | 让出 CPU 给其他进程 |
| **合计帧内工作** | **< 4ms** | 远低于 8ms 预算 |

### 4.3 实际帧率

实际帧率受多方面限制：

| 影响因素 | 对帧率的影响 |
|---------|------------|
| Windows 定时器精度 | 默认 15.6ms 精度，会导致 sleep 超时。需 `timeBeginPeriod(1)` 提升到 1ms |
| 终端模拟器渲染速度 | Windows Terminal 内部约 30~60fps 刷新，即使程序写入 125fps 也可能被合帧 |
| CPU 调度延迟 | 线程唤醒延迟通常 0.5~2ms |
| 输出缓冲区满 | 如果终端来不及消费，write 会阻塞 |

**实际表现：**

| 环境 | 预估实际帧率 | 说明 |
|------|------------|------|
| Windows Terminal | 60~125 fps | 取决于终端 VSync 和渲染管线 |
| cmd.exe | 30~60 fps | 渲染能力较弱 |
| Linux xterm/alacritty | 100~125 fps | GPU 加速终端通常更快 |
| SSH 远程 | 10~30 fps | 受网络延迟限制 |

### 4.4 Windows 定时器精度问题

Windows 默认 `Sleep` / `sleep_until` 精度是 15.6ms（1/64秒）。这意味着 `sleep_until(now + 8ms)` 实际可能睡 15ms。

解决方法（本项目未加，可选优化）：

```c
#include <timeapi.h>
#pragma comment(lib, "winmm.lib")
timeBeginPeriod(1);  // 将定时器精度提升到 1ms
// ... 游戏循环 ...
timeEndPeriod(1);    // 恢复默认
```

加上后 Windows 上的帧定时精度从 ±15ms 提升到 ±1ms。

## 五、完整渲染流程图

```
游戏开始
  │
  ├─ enable_vt100()               开启 Windows ANSI 支持
  ├─ hide_cursor()                隐藏光标
  ├─ printf("\033[?1049h")        切到备用屏幕
  ├─ printf("\033[2J\033[H")      清空备用屏幕并归位
  ├─ fflush(stdout)               确保立即生效
  ├─ next_frame = now()           记录起始时刻
  │
  ╔═══════════════ 每帧循环 ═══════════════╗
  ║                                         ║
  ║  1. 非阻塞读取键盘（4次 get_key_nb）    ║
  ║  2. 音符判定逻辑                        ║
  ║  3. 构建帧缓冲区 buf[]：               ║
  ║     ┌─ \033[H          光标归位         ║
  ║     ├─ 谱面名称行      + \033[K        ║
  ║     ├─ 成绩统计行      + \033[K        ║
  ║     ├─ 空行            + \033[K        ║
  ║     ├─ 判定状态行      + \033[K        ║
  ║     ├─ 顶部边框行      + \033[K        ║
  ║     ├─ 10行游戏区域    + \033[K × 10   ║
  ║     ├─ 判定线边框      + \033[K        ║
  ║     ├─ 按键提示行      + \033[K        ║
  ║     ├─ 退出提示行      + \033[K        ║
  ║     └─ \033[J          清除底部残留     ║
  ║  4. WriteConsoleA / write  一次写出     ║
  ║  5. next_frame += 8ms                   ║
  ║  6. sleep_until(next_frame)             ║
  ║                                         ║
  ╚═════════════════════════════════════════╝
  │
  ├─ printf("\033[?1049l")        切回主屏幕
  ├─ show_cursor()                恢复光标
  ├─ clear                        清屏
  └─ 显示结算画面（cout 正常输出）

游戏结束
```

## 六、ANSI 转义序列速查表

| 序列 | 名称 | 含义 |
|------|------|------|
| `\033[H` | CUP | 光标移到第1行第1列 |
| `\033[K` | EL | 清除光标到行尾 |
| `\033[2K` | EL(2) | 清除整行 |
| `\033[J` | ED | 清除光标到屏幕底部 |
| `\033[2J` | ED(2) | 清除整个屏幕 |
| `\033[?25l` | DECTCEM | 隐藏光标 |
| `\033[?25h` | DECTCEM | 显示光标 |
| `\033[?1049h` | Alt Screen | 切换到备用屏幕缓冲区 |
| `\033[?1049l` | Alt Screen | 切换回主屏幕缓冲区 |
| `\033[n;mH` | CUP | 光标移到第n行第m列 |
| `\033[nA` | CUU | 光标上移n行 |
| `\033[nB` | CUD | 光标下移n行 |

## 七、各方案对比

| 方案 | 闪烁 | 滚动残留 | 撕裂 | 理论帧率上限 | 复杂度 |
|------|------|----------|------|-------------|--------|
| system("cls") + cout | 严重 | 无 | 严重 | ~30 fps | 最低 |
| \033[H + cout 逐行 | 轻微 | 可能 | 中 | ~60 fps | 低 |
| \033[H + buf 单次写 | 无 | 可能 | 极低 | 125+ fps | 中 |
| 备用屏幕 + buf 单次写 | 无 | 无 | 极低 | 125+ fps | 中（本项目） |
| 双缓冲区（自实现） | 无 | 无 | 无 | 200+ fps | 高 |

## 八、可能的进一步优化

| 优化项 | 效果 | 代价 |
|--------|------|------|
| timeBeginPeriod(1) | Windows 定时精度从 15ms 提升到 1ms | 需要链接 winmm.lib |
| 脏区域检测 | 只重绘变化的行，减少写入量 | 代码复杂度增加 |
| 双缓冲 diff | 只输出前后帧差异部分 | 需要维护两帧的完整状态 |
| GPU 加速终端 | 提升终端侧渲染帧率 | 用户需安装特定终端（如 alacritty） |
| 减少帧间隔到 4ms | 理论 250fps | 超出多数终端渲染能力，无意义 |
